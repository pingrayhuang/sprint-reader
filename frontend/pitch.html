這就是 Sprint Reader 要解決的問題：讓業務員在通勤、等候的 7 分鐘內，完成一章複習並立即測驗，系統自動記錄弱點，下次推薦最需要複習的章節。
解法 · 三個核心設計

Sprint Reader 的不同

⏱

7 分鐘限時衝刺

把 110 頁筆記切成 6 章 × 5 張卡。每章 7 分鐘，剛好填滿通勤或等候的空檔，降低「沒時間讀書」的阻力。

🎯

即將測驗觸發

讀完立即接 3 題 quiz，搭配解釋與原文卡片回溯。「即將被考」的心理壓力讓閱讀從被動瀏覽變成主動提取。

🔁

間隔重複推薦

根據「答錯率 × 距上次複習天數」自動推薦最需複習的章節。系統幫你決定「今天要讀哪章」。

技術亮點 01 / 03
偵測業務員「切到 Line 去了」

閱讀中途切換到其他 app 或分頁，計時器自動暫停，切回後從原處繼續。每次切換記一筆 tab_switch_count 到後端。

為什麼這樣設計？
不懲罰業務員被打斷，但記錄下來。tab 切換次數高的使用者，答錯率也更高 — 這是「需要個別輔導」的早期訊號，也是未來 AI 推薦的輸入特徵。

技術亮點 02 / 03
Timer 由後端蓋章，前端無法作弊

所有 timestamp 由後端 session_manager.py 記錄。即使使用者改瀏覽器時間或修改 JS，閱讀時長仍正確。

設計取捨：前端只顯示倒數，不持有真實時間。每個狀態轉換都寫入 SprintSessions，保留完整稽核軌跡，可作為合規完訓證明。
# session_manager.py def start_sprint(module_id, agent_id): sprint_id = str(uuid4()) db_execute( """INSERT INTO SprintSessions (sprint_id, agent_id, module_id, start_timestamp, completion_status) VALUES (?,?,?,?,?)""", (sprint_id, agent_id, module_id, utcnow(), 'in_progress') ) return sprint_id def complete_sprint(sprint_id): db_execute( """UPDATE SprintSessions SET end_timestamp=?, completion_status=? WHERE sprint_id=?""", (utcnow(), 'completed', sprint_id) )
技術亮點 03 / 03
系統自動決定「今天要讀哪章」

每次完成 quiz 後，後端更新 ChapterMastery，根據答錯率與距上次複習天數算出優先分數，推薦最需複習的章節。

公式：
priority = wrong_rate × log(days_since + 1)

答錯越多、越久沒讀 → 分數越高 → 越早被推薦。
未來 Phase 3：升級為強化學習（RL），動態預測每位員工「即將遺忘」的時點。
# services/mastery.py def get_recommendation(agent_id): rows = db_query( """SELECT module_id, wrong_rate, julianday('now') - julianday(last_reviewed_at) AS days_since FROM ChapterMastery WHERE agent_id = ? ORDER BY wrong_rate * log(days_since + 1) DESC LIMIT 1""", (agent_id,) ) return rows[0] if rows else None
