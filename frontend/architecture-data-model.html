📖 0. 先讀這個：資料庫在存什麼？（白話版）
如果你看不懂下面的 schema 表，先看這節。
一句話版本
「使用者（葉悟員）每次開一個章節來讀 7 分鐘，系統會把他讀多久、切去看 Line 幾次、有沒有讀完、有沒有通過 quiz記下來， 並用遺忘曲線演算法自動算出「他下次該什麼時候回來複習這章」。所有紀錄都標好時間戳，未來資料科學家可以拿去做預測模型。」
每張表在做什麼？（用人話）
表名它存的是
Agents🧑 誰是這個使用者（葉悟員、員工編號、部門）。未來接 SSO 後會有很多人。
MicroModules📚 有哪些章節（6 章保險考照重點）。
FlashcardPages📄 每章有哪 5 張卡片（卡片內容）。
QuizQuestions❓ 每章的題庫（題目、選項、正解、解釋）。
SprintSessions ⭐⏱️ 每一次 7 分鐘閱讀的紀錄（誰讀的、讀哪章、幾點開始、分心幾次、有沒有讀完）。本專案的核心行為資料。
QuizResponses📝 每次答題的紀錄（選了哪個、對錯）。
LearningJourney_Map ⭐🔗 把「閱讀」和「測驗」綁在一起的橋樑。同一個人讀完同一章的 quiz，透過這張表串起來算分。
ChapterMastery 🆕🎯 每人每章的「學會程度」總表。存目前的遺忘曲線狀態、下次該複習的時間、是否已精熟。
ReviewEvents 🆕📜 時間軸日誌（像監視器錄影）。每次有動作就多一筆，資料科學家可以 replay 任何時間點。
關係怎麼串起來？
Agents（一個人）
  └─ SprintSessions（他的每次閱讀）  ← 寫進 ChapterMastery（更新他每章的學會程度）
    └─ LearningJourney_Map（連到 quiz）
      └─ QuizResponses（答題紀錄）
    
每一步動作都會多寫一筆 ReviewEvents，所以有完整時間序列。

🔗 1.5. 重點拆解：LearningJourney_Map 那三個規格欄位
這張表是 Project 3 的靈魂。面試官 99% 會問「為什麼要這張表」。
📌 一句話：這張表把「閱讀」和「測驗」綁成同一條學習旅程
沒有這張表，SprintSessions（閱讀紀錄）和 QuizResponses（答題紀錄）就是兩個孤島， 沒人知道「今天答錯這題的人，是哪一次閱讀完就來考的」。規格書講的「Connective Tissue」就是這個概念——結締組織，把兩塊肉連起來。
欄位型別意義（白話）為什麼重要
journey_id 🔑 INTEGER PK 「這條學習旅程」的獨立編號，每一次閱讀 + 測驗組合都有一個。 讓資料科學家可以說「第 37 號 journey 是誰的」，而不用每次 JOIN 兩張表。
sprint_id TEXT FK 指向 SprintSessions.sprint_id——「是哪一次 7 分鐘閱讀」。 JOIN 回去就能查「這次讀花多久、分心幾次、讀哪章」。沒這個，考完試不知道這人到底有沒有先讀。
quiz_session_id TEXT FK 指向 P2 的 QuizResponses.quiz_session_id——「是哪一次測驗」。 JOIN 回去就能查「這次考了幾題、對幾題、哪題錯」。跨 P3 / P2 兩個系統的唯一橋樑。
🎯 面試答題範本
Q：「為什麼需要 LearningJourney_Map 這張表？直接在 QuizResponses 加一個 sprint_id 欄位不就好了？」
A：「因為 P3 和 P2 是兩個獨立 service，規格書明講 P2 是另一個專案。如果把 sprint_id 塞進 P2 的 QuizResponses，等於讓 P2 耦合到 P3 的 schema。 用 Journey_Map 當橋樑，兩邊都只管自己的 PK，橋樑屬於共同的 analytics 層，未來要換掉任一邊都不會斷。 這張表是資料庫版本的 anti-corruption layer。」
# 實戰範例：想查「葉悟員每次 7 分鐘閱讀花多久 + 該次測驗分數」： SELECT s.sprint_id, s.module_id, s.tab_switch_count -- 那次分心幾次, (julianday(s.end_timestamp) - julianday(s.start_timestamp)) * 86400 AS reading_sec, lm.score, -- 自訂欄位：該次考對幾題 lm.total_questions FROM SprintSessions s JOIN LearningJourney_Map lm ON lm.sprint_id = s.sprint_id WHERE s.agent_id = 'demo-agent-001' ORDER BY s.start_timestamp DESC; # 注意：score / total_questions 是我加的（橘色欄位）， # 規格書只要求 journey_id / sprint_id / quiz_session_id（綠色）。 # 我加這兩欄是為了 stats 頁不用再去 QuizResponses JOIN 算分，效能更好。
② 遺忘曲線計算邏輯（SM-2 lite）
每次 quiz 完成 → ChapterMastery 重算 → next_review_at 更新。
每次複習後，ease_factor 依 recall_quality（由 quiz 分數映射 0–5）上下調整；下次最佳複習時間 = 今天 + current_interval_days。分數愈高 → interval 愈長；分數低 → interval 重設為 1 天。
# 映射規則（app/services/mastery.py）： # score_pct >= 100 → quality 5 (完美) # score_pct >= 85 → quality 4 # score_pct >= 70 → quality 3 (通過門檻) # score_pct >= 50 → quality 2 (需重讀) # quality < 3 → interval 重設為 1 天
③ 7 分鐘 timeout 的處理邏輯
規格書規定 completion_status 要能區分 timed_out。UI 端加上提示 + 不納入進度。
使用者路徑：
1. 倒數 7:00 → 0:00 ——tick() 呼叫 handleTimeout()
2. POST /api/sprint/complete with status="timed_out" → SprintSessions.completion_status 寫入 timed_out
3. session_manager.complete_sprint() 檢測到 status ∈ {timed_out, abandoned} → 呼叫 mastery.record_invalid_sprint() → 更新 ChapterMastery.total_timed_out +1 + 新增 ReviewEvent
4. 前端彈出 modal：「偵測到你已超過 7 分鐘尚未完成閱讀…本次紀錄將不納入學習進度」
5. 使用者點「返回首頁」→ location.href = "index.html"
為何 timed_out 不算？ 規格書 Business Context 強調「active reading mode」—— 倒數壓迫使用者主動讀完。如果 timed out session 也計入完成章節，會破壞這個行為激勵機制。stats 層用 WHERE completion_status = 'finished_early' 明確排除。但這些 session 仍保留在 DB（不是刪除），資料科學家可以分析「哪些人常 timeout / 哪章最容易 timeout」。
