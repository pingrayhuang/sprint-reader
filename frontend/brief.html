1. 為什麼要做這個？（The Why）

2. 解法：從「目的地式學習」轉向「心流式學習」

① 章節化衝刺（Flow-state Learning）

把 110 頁《人身保險》筆記切成 6 章 × 7 分鐘包，讓學習融入每日習慣，而不是季度義務。

② Tinder-style 滑卡 UI

左滑下一張、右滑上一張。符合年輕世代 mobile-first 的滑動慣性，降低認知阻力。

③ 間隔重複（Spaced Repetition）

每次做完 quiz，系統根據「答錯率 × 距離天數」算出下一個推薦章節。Day 3 / Day 10 主動推播複習，強迫認知提取。

④ 行為遙測（資料護城河）

記錄 tab_switch_count、reading_time、每題正誤，組成使用者「弱點圖譜」— 未來 RL 演算法的訓練資料。

3. 技術亮點

Browser Visibility API：偵測「業務員切到 Line」

監聽 document.visibilityState，切走時 timer 暫停、伺服器 tab_switch_count++，切回後恢復。不只是 UX 設計，更是行為數據：tab 切換次數高的業務員，未來測驗錯誤率顯著升高，可作為「需個別輔導」的早期訊號。

嚴格狀態機：Server-authoritative timing

所有 timestamp 由後端 session_manager.py 蓋章，前端只負責顯示。即使業務員改瀏覽器時間或 JS，紀錄仍正確。狀態機：pre_sprint → reading → handoff，每個 transition 都寫入 SprintSessions。

4. MA 跨部門價值

這個專案涵蓋四個 function，每個面向都有對應的商業意義：

Function對話內容本專案的角色
人資 / 教育訓練部「新人考照通過率要怎麼拉高？」6 章結構化複習、弱點圖譜
業務管理部「哪些新人要額外輔導？」答錯率 + tab 切換次數 → 風險分級
資料部 / AI「行為數據能幹嘛？」LearningJourney_Map → 個人化推薦、自動出題
數金部 / IT「上線複雜度多高？」FastAPI + SQLite，零依賴一鍵啟動

上線後延伸：考完照後可轉成「在職續訓工具」— 新法規一出，立刻切成 7 分鐘包推播給所有業務員，取代冗長的年度回訓。

5. Roadmap（如果這成為凱基的真實產品）

  1. Phase 1（已完成）：MVP — 7 分鐘滑卡 + 隨機 3 題 quiz + 規則式間隔推薦
  2. Phase 2（3 個月）：接入凱基真實合規文件與訓練教材，用 LLM 自動把長文切成 7 分鐘 sprint 包
  3. Phase 3（6 個月）：把 wrong_rate × days_since 的規則式推薦升級為 強化學習（RL），動態預測每位員工「即將遺忘」的時點，在最佳複習時機推播
  4. Phase 4（12 個月）：行為訊號接入「業務員風險評分」，提早預警違規高風險個案，建立個人化學習推薦引擎

7. 我從這個專案學到什麼

  • 產品思維：設計時主動把「資料如何被下游用」想進去，而不是只做眼前的 feature
  • 跨部門語言：同一個產品，能對教育訓練、法遵、資料、IT 講出對方關心的 KPI
  • 商業洞察：把「合規」這個看似 cost center 的工作，重新框架成「風險預警資產」
  • 技術選型：用最小成本（無 framework、無 build step）做出可 demo 的完整流程
